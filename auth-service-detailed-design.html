1. 背景与设计目标

`aihub` 与 `aipipeline` 当前都包含用户、登录、JWT、系统成员和权限控制相关能力。两个系统的业务域不同，但身份域正在趋同，适合抽出统一的身份权限中心。

设计目标

统一登录、用户、系统、成员、全局角色、系统角色和权限视图，支持 JWT 本地验签和后续公司统一认证平台接入。

非目标

不接管 Skill/Prompt 审批、Nacos、流水线执行、模板 scope、节点状态等业务资源规则。

统一认证统一鉴权系统成员SSO ReadyJWT / JWKS

3. 服务职责边界

auth-service 负责

  • 登录、登出、刷新 Token、JWKS 发布
  • 用户资料、账号状态、外部身份映射
  • 全局角色、系统、系统成员、系统角色
  • 权限视图、鉴权 API、管理审计

业务服务保留

  • `aihub` 保留资源 owner、审批状态、Nacos namespace
  • `aipipeline` 保留 pipeline/template scope、执行状态、节点规则
  • 业务数据库不被 `auth-service` 直接访问

5. 登录与鉴权链路

sequenceDiagram participant User as 用户 participant GW as 网关 participant AuthFE as auth 前端 participant AuthBE as auth 后端 participant UIAS as 公司统一认证 participant AihubFE as aihub 前端 participant AihubBE as aihub 后端 participant PipeFE as aipipeline 前端 participant PipeBE as aipipeline 后端 User->>GW: 访问 aihub 或 aipipeline GW->>AuthFE: 未登录，跳转统一登录页 AuthFE->>AuthBE: 发起登录 AuthBE->>UIAS: 后续接入公司统一认证 UIAS-->>AuthBE: 返回企业用户身份 AuthBE->>AuthBE: 创建或更新本地影子用户 AuthBE-->>AuthFE: 返回 Access Token / Refresh Token AuthFE-->>User: 登录完成 User->>GW: 打开 aihub GW->>AihubFE: 返回 aihub 前端 AihubFE->>AihubBE: 调用业务 API，携带 Bearer Token AihubBE->>AuthBE: JWKS 验签或查询权限视图 AuthBE-->>AihubBE: 返回用户、系统、角色、权限 AihubBE-->>AihubFE: 返回资源数据 User->>GW: 打开 aipipeline GW->>PipeFE: 返回 aipipeline 前端 PipeFE->>PipeBE: 调用流水线 API，携带 Bearer Token PipeBE->>AuthBE: JWKS 验签或查询权限视图 AuthBE-->>PipeBE: 返回用户、系统、角色、权限 PipeBE-->>PipeFE: 返回流水线数据
  1. 用户访问 `aihub` 或 `aipipeline`，未登录时跳转 `auth-service`。
  2. 第一阶段使用本地账号；后续由 `auth-service` 对接公司统一认证平台。
  3. 认证成功后签发 Access Token 与 Refresh Token。
  4. 业务后端通过 JWKS 本地验签，必要时调用 `/api/authz/check`。
  5. 业务后端继续校验资源归属、scope、owner、审批状态、执行状态等本地规则。

6. Token 设计

JWT 只放稳定且轻量的信息，不把完整系统成员列表塞进 Token。

{
  "iss": "http://auth-service.internal",
  "sub": "u_001",
  "username": "alice",
  "globalRole": "USER",
  "source": "SSO",
  "aud": ["aihub", "aipipeline"],
  "iat": 1710000000,
  "exp": 1710001800
}
项目建议说明
Access Token30 分钟业务 API 使用，短期有效。
Refresh Token14 天用于静默刷新，可服务端撤销。
JWKS Cache5 到 15 分钟业务后端缓存公钥。
Permission View Cache1 到 5 分钟用户系统角色视图缓存。

9. 与 aihub / aipipeline 集成

系统改造点保留本地规则
aihub 登录入口改为 `auth-service`；后端改为 JWKS 验签；用户搜索、系统列表、成员管理迁往 `auth-service`。 Skill/Prompt/MCP/Agent 的 owner、approval、namespace、resource scope。
aipipeline 登录入口改为 `auth-service`；`SysUser`、`BizSystem`、`SystemMember` 迁入或改为只读镜像。 Pipeline、Template、Node 的 `ipmpSystemNum`、scope、执行状态和资源状态。
服务间调用使用 `client_credentials`、mTLS 或网关内网白名单；常用权限视图可缓存，敏感变更通过版本号或事件通知失效。

10. 公司统一认证平台接入

UIAS，在 `auth-service` 内部新增 provider adapter，业务系统无感知。

能力设计
影子用户企业认证成功后，根据外部唯一标识查找或创建本地用户。
映射优先级`provider + external_subject`，其次 employeeId、email，最后人工绑定。
默认权限首次登录默认 `USER`，不自动授予 `SYSTEM_ADMIN`。
故障兜底保留本地平台管理员账号，用于统一认证故障应急。

11. 安全与部署

安全控制

  • Access Token 非对称签名，业务服务只持有公钥。
  • Refresh Token 服务端保存 hash，支持撤销。
  • 管理员操作强制审计，记录 traceId 和 reason。
  • 防止自删、自降级、最后系统管理员删除。

部署建议

  • `auth-service` 后端无状态多副本部署。
  • `auth_db` 使用主备或托管高可用。
  • Redis 可用于权限视图缓存和 Token 黑名单。
  • 密钥挂载或接入内部 KMS。

12. 迁移计划

阶段目标关键动作
一统一模型和文档建立工程，固化用户、系统、成员、角色、接口设计。
二抽只读权限中心导入用户、系统、成员数据，业务系统开始读取权限视图。
三统一登录入口前端跳转统一登录页，业务后端改为 JWKS 验签。
四接入公司统一认证新增 provider adapter，建立外部身份映射。
五收敛本地用户表业务系统用户表改为只读缓存、审计冗余字段，或完全移除。

13. 测试与验收

类别验收点
认证登录、刷新、登出、Token 撤销、JWKS key rotation。
权限普通成员、系统管理员、平台管理员权限符合矩阵。
集成`aihub` 与 `aipipeline` 携带同一 Token 调用业务 API。
变更用户禁用、角色调整、成员移除后权限视图按预期失效。

14. 风险与应对

风险应对
权限中心不可用影响业务JWT 本地验签、权限视图短期缓存、只在敏感操作回源。
角色变更生效延迟权限版本号、缓存短 TTL、变更事件通知。
历史用户数据不一致先做数据盘点和映射表，迁移时保留 username 与 external subject。
公司统一认证字段不稳定通过 `external_identities` 做 provider 适配，不把外部字段散落业务系统。

15. 设计结论

`auth-service` 应作为统一身份权限中心独立部署。它集中管理用户、系统、成员、角色、Token、权限视图和公司统一认证接入；`aihub` 与 `aipipeline` 保留各自业务资源和资源状态校验。

这条边界既能解决重复用户权限体系的问题，又能避免把业务规则全部堆进认证中心。后续接入公司统一认证平台时，只需要改造 `auth-service` 的认证适配层，两个业务系统基本保持稳定。
